# v0.5 BrowserUser 爬虫智能体补充方案

_状态：规划中（2026-06-18）_

## 背景

1000 家评估证明 CNINFO 年报 PDF 程序化抽取可行（92.4% strict-usable），但存在三类 PDF 无法覆盖的数据：

1. **PDF 中不存在的信息**：投资者互动问答、官网最新业务动态
2. **PDF 中格式不适配**：金融公司按工业 schema 抽取
3. **PDF 抽取质量不足**：rnd 数值误抓、表格 header-only

BrowserUser 定位为**补充**而非替代：只在 PDF 抽取无法覆盖或质量不足的场景介入。

## 目标

1. 验证 BrowserUser 在 1–2 个目标数据源上的可行性
2. 产出与 `company_profile.json` 格式兼容的结构化记录
3. 不影响现有 CNINFO 年报主链路

## 候选数据源（按优先级）

### P0：投资者互动平台

| 属性 | 说明 |
|---|---|
| 平台 | 上证 e 互动（sns.sseinfo.com）、深交所互动易（irm.cninfo.com.cn） |
| 数据 | 投资者问答（公司回复） |
| 价值 | 业务细节、产能、客户、竞争等 PDF 未披露信息 |
| 难点 | 需搜索/翻页/JS 渲染 |
| BrowserUser 优势 | 可模拟搜索 → 翻页 → 提取问答 |

### P1：公司官网 IR 页面

| 属性 | 说明 |
|---|---|
| 数据 | 业务介绍、产品列表、最新公告 |
| 价值 | 补充 major_products / main_business_segments |
| 难点 | 每家公司 IR 页面结构不同 |
| BrowserUser 优势 | 可适应不同页面布局 |

### P2：交易所非年报公告

| 属性 | 说明 |
|---|---|
| 数据 | 重大事项、关联交易、增减持 |
| 价值 | 事件信号 |
| 难点 | CNINFO API 已部分覆盖；BrowserUser 用于 API 未索引的 |
| 备注 | 先评估 API 是否足够 |

### P3：政府采购 / 招投标

| 属性 | 说明 |
|---|---|
| 数据 | 中标信息 |
| 价值 | 下游客户/订单线索 |
| 难点 | 搜索接口 + 分页 |
| 备注 | 公开但结构不统一 |

## 不做的事

- **不替代 CNINFO 年报抽取**：已有 946/1020 稳定链路
- **不爬取强反爬/付费源**：企查查、Wind 等
- **不存新闻/研报全文**：仅标题 + 链接 + 摘要
- **不用 LLM 做字段抽取**：BrowserUser 负责「获取页面内容」，抽取仍用规则

## 技术方案（草案）

```
BrowserUser Agent
    │
    ▼
目标 URL + 任务描述（如「搜索公司 X 在 e 互动的最近 20 条问答」）
    │
    ▼
BrowserUser 执行（搜索/翻页/点击/提取）
    │
    ▼
原始 HTML / 文本
    │
    ▼
规则解析 / 结构化
    │
    ▼
supplement_profile.json（与 company_profile.json 格式对齐）
    │
    ▼
合并入库（标注数据来源 = browser_agent）
```

### 输出格式

```json
{
  "field": "investor_qa",
  "source": "browser_agent",
  "source_url": "https://sns.sseinfo.com/...",
  "fetch_time": "2026-06-18T...",
  "status": "found",
  "value": [{"question": "...", "answer": "...", "date": "..."}],
  "evidence_sentence": "..."
}
```

### 与现有 pipeline 的接口

- 新增 `collectors/browser_agent_collector.py`（或在 `lab/` 下独立模块）
- 输入：company list + target source + task template
- 输出：`outputs/supplements/<code>/supplement_profile.json`
- 合并：按 company_code join 到主 profile，字段带来源标注

## 试点计划

### 第一步：e 互动单公司验证

- 选 3 家公司（CATL、三一、招商银行）
- BrowserUser 搜索并提取最近 10 条问答
- 人工检查：内容是否有价值？格式是否可结构化？
- 预计耗时：1–2 天

### 第二步：10 公司批量试点

- 如果第一步成功，扩到 eval1000 中 10 家
- 对比 PDF 抽取 vs BrowserUser 补充的字段覆盖
- 评估 BrowserUser 成本（调用次数 × 单价）

### 第三步：决策

- 如果 ROI 合理 → 纳入 ROADMAP 第四阶段
- 如果成本过高或质量不稳定 → 降级为手动补充

## 风险

| 风险 | 缓解 |
|---|---|
| 页面结构变化 | 任务模板 + 错误重试；不 hard-code 选择器 |
| BrowserUser 成本高 | 先小样本试点，评估 ROI |
| 合规 | 仅公开页面；不登录；遵守 ToS |
| 数据质量 | 保留原始 HTML + 人工 spot-check |

## 前置条件

- v0.4 1000 家评估完成（已完成）
- rnd / 表格规则收紧（建议先做，减少 BrowserUser 需补充的字段）
- BrowserUser 环境可用
